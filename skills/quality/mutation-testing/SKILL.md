# Skill: Mutation Testing (Infection)

## Контекст

Мутационное тестирование измеряет **качество** тестов, а не только покрытие. Infection вносит маленькие изменения (мутации) в исходный код — заменяет `>` на `>=`, `+` на `-`, удаляет вызов метода, инвертирует условие — и перезапускает тесты. Если после мутации тесты падают — мутант «убит» (kill), тесты заметили поломку. Если тесты по-прежнему зелёные — мутант «выжил» (escaped): строка покрыта, но ни один ассерт реально не проверяет её поведение.

**Когда активировать:**
- Настраивают Infection в PHP-проекте (создают/правят `infection.json5`).
- Добавляют мутационный гейт качества тестов в CI (отдельный workflow или шаг).
- Читают отчёт Infection — нужно понять MSI / Covered MSI / escaped mutants.
- Разбираются, почему мутанты выживают, и куда дописать ассерты.
- Выбирают, что исключить из мутирования (Commands, Facades, миграции).

**Граница со скиллом `quality/test-strategy`:** там — стратегия (пирамида, какой уровень тестов нужен фиче, политика coverage). Здесь — конкретный инструмент Infection: конфиг, запуск, гейты, чтение отчёта. Если вопрос «сколько и каких тестов писать» — это `test-strategy`; если «как настроить/прочитать Infection» — этот скилл.

## Предусловия

1. **Покрытие в формате XML.** Infection требует line-coverage. Тестам нужен драйвер покрытия (Xdebug или PCOV) и прогон с `--coverage-xml`. Без покрытия Infection не знает, какие строки мутировать с тестами.
2. **Зелёный набор тестов.** Мутационное тестирование осмысленно только когда базовый прогон зелёный — иначе все мутанты «уже убиты» поломанным набором.
3. **Установлен Infection:** `composer require --dev infection/infection`. Для пакетов-моно с несколькими `src` — один корневой `infection.json5` с несколькими `directories`.

## Алгоритм

1. **Создать `infection.json5`** в корне проекта. Ключевые блоки (см. `snippets/infection.json5`):
   - `source.directories` — каталоги исходников. В монорепо/пакете перечисляются все `packages/*/src`.
   - `source.excludes` — что НЕ мутировать (об этом ниже, секция «Исключения»).
   - `testFramework` — `pest` или `phpunit`.
   - `mutators` — профиль мутаторов: `{"@default": true}` как база, точечно добавляются `UnwrapArrayFilter`, `MethodCallRemoval` и т. п.
   - `logs` — куда писать отчёты (`text`/`html`/`summary`/`json`).
   - `minMsi` / `minCoveredMsi` — гейты (см. «Гейты»).
2. **Прогнать локально для базовой линии:** сначала сгенерировать покрытие, затем запустить Infection по этому покрытию:
   ```bash
   vendor/bin/pest --coverage-xml=build/coverage/xml --min=0
   vendor/bin/infection --threads=4 --coverage=build/coverage/xml --no-progress
   ```
   `--threads=4` (или `--threads=max`) распараллеливает мутантов — без этого Infection очень медленный. Переиспользование готового `--coverage` экономит повторный прогон тестов.
3. **Прочитать отчёт.** Главная метрика — **MSI (Mutation Score Indicator)**. Открыть `summary`/`html`, найти escaped mutants (см. «Интерпретация»).
4. **Зафиксировать гейты.** Поднять `minMsi`/`minCoveredMsi` в `infection.json5` до достигнутого уровня минус небольшой буфер — чтобы качество тестов не деградировало в будущем.
5. **Включить в CI.** Добавить шаг/workflow с инкрементальным режимом для PR и/или полным прогоном на merge в main (см. `snippets/mutation-ci.yml` и секцию «CI-гейт»).
6. **Итерировать по выжившим мутантам.** Для каждого escaped mutant: либо дописать ассерт, который убивает мутацию, либо осознанно проигнорировать (`@infection-ignore-all` / `mutators` отключить точечно), если мутация эквивалентна.

## infection.json5: ключевые поля

| Поле | Назначение |
|:---|:---|
| `source.directories` | Каталоги исходников для мутирования; в пакетах — список всех `packages/*/src` |
| `source.excludes` | Глобы/имена каталогов, исключённые из мутаций (Commands, Facades, миграции, Filament) |
| `testFramework` | `pest` или `phpunit` |
| `testFrameworkOptions` | Доп. флаги фреймворка (напр. генерация `coverage-xml`, `junit`) |
| `mutators` | Профиль мутаторов: `@default` + точечные включения/выключения |
| `logs.text` / `.html` / `.summary` / `.json` | Форматы отчётов: `text` — список escaped, `html` — навигация, `summary` — агрегаты, `json` — для машинной обработки |
| `timeout` | Таймаут на одного мутанта (сек); защищает от бесконечных циклов мутаций |
| `threads` | Параллелизм по умолчанию (флаг `--threads` перекрывает) |
| `minMsi` | Гейт по общему MSI — падает CI если ниже |
| `minCoveredMsi` | Гейт по MSI только покрытых строк — падает CI если ниже |

**Профиль мутаторов.** Старт — `{"@default": true}`: включает все «безопасные» мутаторы. Точечно добавляются специфичные (`UnwrapArrayFilter`, `MethodCallRemoval`), а шумные/эквивалентные — выключаются (`"MutatorName": false`). Не включать всё подряд: лишние мутаторы дают эквивалентные мутации (которые невозможно убить) и роняют MSI без пользы.

## Гейты: minMsi vs minCoveredMsi

- **MSI** = `(killed + timeout + error) / total_mutants`. Учитывает ВСЕ мутанты, включая те, что в непокрытом коде (они автоматически «не убиты»). Низкий покрытие → низкий MSI.
- **Covered MSI** = тот же расчёт, но знаменатель — только мутанты в **покрытых** строках. Отвечает на вопрос: «из того, что тесты вообще трогают, какую долю они реально проверяют?»
- **Зачем оба:** `minMsi` ловит общий разрыв (включая непокрытый код), `minCoveredMsi` — строже к качеству существующих тестов. Типовая планка: `minMsi: 70`, `minCoveredMsi: 80` — Covered выше, потому что от покрытого кода ждём более качественных ассертов.
- Гейты дублируются: в `infection.json5` (для локального запуска) и/или флагами `--min-msi` / `--min-covered-msi` в CI. Флаг перекрывает конфиг.

## Интерпретация: MSI и escaped mutants

- **Killed** — тесты упали после мутации. Хорошо: поведение проверяется.
- **Escaped** — тесты прошли несмотря на мутацию. Дыра в ассертах: строка выполняется, но результат не проверяется. Главная цель ревизии.
- **Not Covered** — мутант в непокрытой строке; тесты её не выполняют вовсе. Это вопрос покрытия (см. `test-strategy`), не качества ассертов.
- **Timeout / Error** — мутация привела к зацикливанию/фатальной ошибке; засчитывается как «убит» (тесты бы это поймали).
- **Эквивалентный мутант** — мутация не меняет наблюдаемое поведение (напр. `<` → `<=` на границе, недостижимой тестом по бизнес-смыслу). Убить нельзя; помечают `@infection-ignore-all` или отключают мутатор точечно. Не гнаться за 100% MSI — последние проценты часто эквивалентны.

**Разбор escaped:** открыть `logs.text` или HTML, найти `escaped` мутант → понять, какое изменение пережили тесты → дописать ассерт, который провалится на мутации (проверять конкретное значение/ветку, а не «не упало»). Escaped мутант — почти всегда сигнал слабого ассерта (`assertTrue($x)` вместо `assertSame(3, $x)`).

## Исключения: что не мутировать

Исключают код, где мутации дают шум, а не сигнал (`source.excludes`):

- **Commands / Console** — точки входа CLI; логика должна быть в сервисах, которые и тестируются. Мутации в декларации команды бесполезны.
- **Facades** — тонкие статические прокси; мутировать нечего, тесты на фасад избыточны.
- **Миграции / Database** — детерминированные DDL-скрипты, не покрываются юнит-ассертами.
- **Сгенерированный код / UI-слой админки (Filament и т. п.)** — фреймворочные декларации, мутации эквивалентны.

Правило: исключай **точки входа и декларативный/сгенерированный код**, оставляй **доменную логику** (сервисы, политики, value-объекты) — именно там мутанты находят слабые тесты.

## CI-гейт качества тестов

Мутационное тестирование дорогое (прогон всего набора на каждого мутанта). Стратегия запуска в CI (см. `snippets/mutation-ci.yml`):

1. **Инкремент на PR** — мутировать только изменённые строки: `--git-diff-lines --git-diff-base=origin/main`. Быстро, держит планку на новом коде, не блокируя PR полным прогоном.
2. **Полный прогон на merge в main / по расписанию / manual** — advisory (информативный), не блокер: дорого гонять весь набор на каждый PR.
3. **Покрытие генерируется отдельным шагом** (`--coverage-xml`), затем переиспользуется Infection через `--coverage=` — чтобы не прогонять тесты дважды.
4. **Гейты флагами:** `--min-msi=70 --min-covered-msi=80 --no-progress`. На инкременте — те же планки применяются к диффу.
5. **Драйвер покрытия:** в CI включить Xdebug/PCOV (`coverage: xdebug`, `XDEBUG_MODE=coverage`) — иначе нет line-coverage и Infection не запустится.

Для инкремента нужна полная история git (`fetch-depth: 0`) и доступная база сравнения (`origin/main`), иначе `--git-diff-lines` не найдёт diff.

## Чеклист качества

- [ ] `infection.json5` в корне: `source.directories` покрывают все `src`, `excludes` отсекают Commands/Facades/миграции/UI
- [ ] Профиль мутаторов начат с `@default`; лишние/эквивалентные мутаторы выключены точечно, а не «всё подряд»
- [ ] Заданы `logs` (минимум `text` + `summary`) — есть откуда читать escaped mutants
- [ ] Гейты `minMsi` и `minCoveredMsi` зафиксированы (covered ≥ общий), не ниже достигнутой базовой линии
- [ ] Локальный запуск идёт по готовому `--coverage`, с `--threads` (не однопоточно)
- [ ] CI: инкремент `--git-diff-lines` на PR; полный прогон на main — advisory, не блокер
- [ ] В CI включён драйвер покрытия и `fetch-depth: 0` для diff-режима
- [ ] Escaped mutants разобраны: дописаны точные ассерты или помечены эквивалентными (`@infection-ignore-all`), а не оставлены молча
