# Session Handoff

## Когда активировать

- Контекст сессии близок к лимиту (или предстоит компакция).
- Работа передаётся другому агенту (Claude → Cursor, локальный → CI).
- Сработал жёсткий порог `anti-drift` — сессию нужно завершить и начать заново.
- Конец рабочего дня на многодневной задаче.

## Протокол

1. **Собрать состояние.** `git status` + `git diff --stat`: что изменено,
   что закоммичено, что в работе. Не по памяти — по факту из репозитория.
2. **Зафиксировать решения.** Каждое принятое решение — с обоснованием
   («выбрали X, потому что Y; рассмотрели Z — отвергли из-за W»).
   Решение без «почему» бесполезно для следующего агента.
3. **Выписать незакрытое.** Открытые вопросы, известные баги, отложенные
   правки — с указанием, что блокирует каждую.
4. **Следующие шаги.** Конкретные действия в порядке приоритета,
   с путями к файлам и командами верификации.
5. **Проверка чтением с нуля.** Перечитай документ глазами агента без
   контекста: всё ли понятно без этой сессии? Кодовые имена и сокращения,
   введённые в сессии, — расшифровать или убрать.

## Структура документа

Шаблон — `snippets/handoff-template.md`. Сохраняйте в проекте по
конвенции проекта (например `docs/handoff/YYYY-MM-DD-<topic>.md`)
или передавайте в первом сообщении новой сессии.

## Анти-паттерны

- Писать summary по памяти, не сверяясь с `git status`/`git diff`.
- Перечислять сделанное без «почему» у решений.
- Тащить в handoff весь лог сессии — это сжатие, а не стенограмма.
