# Skill: Code Review Guide

Применять когда: команда > 1 человека, есть git workflow с PR. Не одноразово — это процесс, который кодифицируется один раз и пересматривается раз в квартал. Запускать при формировании команды или после серии плохих merge'ей.

Не применять: solo-разработка (бессмысленно, лучше парная сессия), prototype-стадия без публикации.

---

## Когда НЕ применять

- Solo-разработчик без планов нанимать
- Hotfix critical bug в проде (быстро мержим + post-merge review)
- Авто-сгенерированный код (миграции, OpenAPI client) — проверять только diff схемы, не сам код

---

## Шаг 1. Что обязательно проверять (priority order)

| Приоритет | Категория | Примеры |
|:---|:---|:---|
| **P0** | Безопасность | SQL injection, hardcoded secrets, broken auth, отсутствие input validation |
| **P0** | Корректность | Логика fails на edge-case, race condition, off-by-one, неверный SQL |
| **P1** | Тесты | Нет покрытия для новой логики, удалены тесты без объяснения |
| **P1** | Производительность | N+1 запросы, отсутствие индекса, синхронный IO в hot path |
| **P2** | Архитектура | Нарушение слоёв, циклические зависимости, утечка домена в адаптер |
| **P2** | Читаемость | Магические числа, плохие имена, функция > 50 строк, глубокая вложенность |
| **P3** | Стиль | Форматирование, naming convention — должно ловиться линтером, не глазами |

**Правило:** P3 — задача линтера, не reviewer'а. Если такие комментарии частые → не настроен pre-commit hook.

---

## Шаг 2. Severity градация комментариев

Использовать префиксы — это критично для ИИ-агентов и для скорости автора:

| Префикс | Значение | Действие автора |
|:---|:---|:---|
| **`[blocker]`** | Не мержим пока не исправлено | Обязательно исправить или обосновать отказ |
| **`[issue]`** | Важно, но можно отдельным PR | Исправить сейчас или создать ticket |
| **`[nit]`** | Мелочь / стиль | На усмотрение автора, можно проигнорировать |
| **`[question]`** | Не понял, объясни | Ответить, при необходимости — добавить комментарий в код |
| **`[praise]`** | Хорошее решение | Ничего, но motivation booster |
| **`[suggestion]`** | Альтернатива на подумать | На усмотрение |

Без префикса — комментарий считается `[issue]` по умолчанию.

## Шаг 3. Async vs Sync — когда какой

| Режим | Когда | Плюсы | Минусы |
|:---|:---|:---|:---|
| **Async (комменты в PR)** | По умолчанию | Не отвлекает, можно подумать | Медленнее, легко застрять в пинг-понге |
| **Sync (звонок / парное ревью)** | Сложный PR > 500 строк, junior-автор, архитектурное изменение | Быстро, обучение | Тратит время обоих |
| **Pre-PR sync** | Большая фича, спорный подход | Ловим проблемы до написания кода | Требует дисциплины |

**Правило 2 кругов:** если после 2 кругов async-комментов спор не решён → переход в sync. Дальше — эскалация tech lead.

## Шаг 4. SLA на review

| Размер PR | Целевой SLA | Жёсткий потолок |
|:---|:---|:---|
| < 50 строк | 2 часа | 1 рабочий день |
| 50–200 | 4 часа | 1 рабочий день |
| 200–500 | 1 день | 2 дня |
| > 500 | **разделить PR** | — |

**Анти-паттерн:** PR > 500 строк. Reviewer теряет внимание после ~400, остаток не читается = rubber stamp. Требовать разбивки.

---

## Шаг 5. Author Responsibilities (до запроса review)

Чек-лист автора перед `Request Review`:

- [ ] PR description: что/зачем/как тестировал
- [ ] Скриншоты для UI-изменений
- [ ] Self-review через diff на GitHub (свежим взглядом)
- [ ] CI зелёный (lint + tests)
- [ ] Тесты для новой логики добавлены
- [ ] Migration / breaking change — отмечен в description
- [ ] PR < 500 строк (или объяснение почему нельзя разбить)

Если хотя бы одно — нет, reviewer вправе закрыть с комментом «дополни и пере-открой».

---

## Шаг 6. Reviewer Responsibilities

- Не блокировать на `[nit]` — только `[blocker]` блокирует
- Approve = «я готов отвечать за этот код вместе с автором»
- Если не разбираешься в области (БД-миграция, ML, security) — позвать эксперта, не approve вслепую
- Запрещено `LGTM` без чтения кода. Минимум: посмотрел diff, прочитал тесты, проверил один сценарий мысленно
- На junior-PR — обучающие комментарии со ссылками на доку, не «исправь и всё»

---

## Что агент добавляет сам

- Шаблон PR description (что / зачем / как тестировал / breaking changes / screenshots)
- Шаблон комментариев с примерами фразировок для частых ситуаций
- Метрики ревью: avg time to first review, avg rounds per PR, % PR > 500 строк — для пересмотра процесса
- Эскалация для merge без approve (только tech lead, с записью причины)

## Жёсткие запреты

НЕЛЬЗЯ:
- Approve без чтения кода (rubber stamp)
- Блокировать merge на `[nit]` / стиле / форматировании
- Self-approve (даже если ты единственный senior — позвать второго)
- Игнорировать `[blocker]` без письменного обоснования
- Мержить PR с красным CI (даже «потом починим»)
- PR > 500 строк без явного разрешения tech lead
- Review «по диагонали» PR с security/payments/auth изменениями
