# Сценарии работы с агентом (сценарий → скилл)

Практические сценарии: что попросить у агента в чате и какие **skills** из `.ai/skills/` задействовать (явно в промпте или через контекст правил после генерации). Это не отдельные «агенты-персонажи», а согласованные наборы инструкций. Имена доменных скиллов — примеры; подставьте скиллы вашего проекта.

## Карта сценариев

| Сценарий | С чего начать | Цепочка скиллов |
|---|---|---|
| Ревью PR или диффа | `code-review` | → доменный скилл (`<domain>-workflow`) → `cross-layer-change-checklist` |
| Добавить/поправить тесты | `testing-rules` | → `pest-testing` → pre-flight окружения |
| Security pass | базовый security-скилл | → секция безопасности в `code-review` |
| Кросс-слойное изменение | `cross-layer-change-checklist` | → `complex-task-orchestrator` |
| Git и коммиты | `git-commit-rules` | → preflight для hooks на хосте |

## Ревью кода

**Когда:** перед merge, большой PR, проверка Action/Policy/контроллеров.

**Что сделать:** активировать контекст skill **`code-review`** (упомянуть в запросе «следуй skill code-review» или прикрепить `.ai/skills/code-review/SKILL.md`).

**Цепочка:** архитектура слоёв → при изменении домена — доменный скилл проекта (`<domain>-workflow`) → при нескольких слоях — **`cross-layer-change-checklist`**. Для очень крупных задач дополнительно **`complex-task-orchestrator`**.

## Генерация и прогон тестов

**Когда:** нужны новые тесты, падение CI, проверка доступа через HTTP.

**Порядок:**

1. **`testing-rules`** — где запускать (Docker vs хост), только отдельная тестовая БД (`<app>_test`), опасные команды вне testing-окружения запрещены.
2. **`pest-testing`** — синтаксис Pest, Feature vs Unit, команды `php artisan test`.
3. Pre-flight перед запуском: проверить окружение (контейнеры, БД, зависимости).

Для доменных фич — типично Feature-тесты в `tests/Feature/<Domain>/`.

## Security pass

**Когда:** новые эндпоинты, загрузки файлов, смена прав доступа.

**Порядок:**

1. Базовый security-скилл (практики Laravel: Policy/Gate, валидация, секреты только через `config/*`).
2. В ревью фичи — секция безопасности в **`code-review`**; для загрузок медиа — скилл работы с медиабиблиотекой.
3. При необходимости — доменный скилл для согласованности permission и policy.

## Кросс-слойное изменение

**Когда:** затронуты backend + авторизация + маршруты + фронт + тесты.

**Порядок:**

1. **`cross-layer-change-checklist`** — обязательная цепочка по слоям.
2. При сложной постановке — **`complex-task-orchestrator`** (декомпозиция, DoD, минимальный test-gate).
3. Workflow домена — dependency map в `.ai/guidelines/` и доменный скилл (`<domain>-workflow`).

Frontend-маршруты и типы (enum/DTO) — по контрактам проекта (например, генерация в `resources/types/`).

## Git и коммиты

Для сообщений коммитов и husky/commitlint — **`git-commit-rules`**. Перед коммитом, если нужны `pnpm`/hooks на хосте — preflight-скилл для Node/pnpm.
