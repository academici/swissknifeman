# MCP Blender — инструменты и детали камеры

## MCP-инструменты

| Инструмент | Назначение | Когда использовать |
|---|---|---|
| `execute_blender_code` | Выполнить Python в Blender | основной инструмент для построения и настройки |
| `get_viewport_screenshot` | Скриншот вьюпорта | только после установки камеры (Шаг 6→7) |
| `get_object_info` | Bounding box, вершины, материалы | Шаг 5 — проверка каждой изменённой детали |
| `get_scene_info` | Список всех объектов сцены | см. ниже |

**`get_scene_info` — когда вызывать:**
- **После `setup_scene()`** в инкрементальном режиме — убедиться что старые `MESH_*` объекты удалены и коллекция пересоздана чисто
- **После boolean DIFFERENCE** — проверить что временный инструмент (имена начинаются с `_`) исчез из сцены (если остался — `apply_bool` не отработал)
- **При неожиданном поведении** — понять что вообще есть в сцене перед отладкой

Polyhaven / Sketchfab / Hyper3D — не для parametric-моделей.

## Камера: универсальный принцип

```python
from {CURRENT_MODEL}.common import set_viewport_camera

# z_mm  — центр изменённой зоны в мм (из bounding box Шага 5)
# distance — высота_зоны_мм × 0.001 × 1.5, минимум 0.05
set_viewport_camera(z_mm=<z_centre>, distance=<distance>)
set_viewport_camera()  # вся модель целиком (distance по умолчанию из model_config)
```

- `z_mm=None` → центр: `height/2 - camera_ref_h` (из `model_config`)
- `distance` — чем меньше, тем крупнее план
- Ракурс: `ORTHO`, вид на разрез `X=0`, наклон ~15° (`Euler(75°, 0°, -90°)`)
- **ЗАПРЕЩЕНО** задавать `view_rotation` вручную через Quaternion/Euler — модель перевернётся. Только `set_viewport_camera()`.

Готовые значения `z_mm` / `distance` для конкретной модели — в `models/{CURRENT_MODEL}/ai/rules.md`.

## Режимы отображения вьюпорта

Именованные пресеты. Применять по команде пользователя или самостоятельно выбирать оптимальный для задачи.

### `tech` — технический вид (дефолтный для проверки геометрии)

**Когда:** проверка формы, размеров, стыков, пазов, резьбы. Максимальная читаемость рёбер и полостей.

```python
import bpy
bpy.ops.object.select_all(action='DESELECT')
for area in bpy.context.screen.areas:
    if area.type == 'VIEW_3D':
        for space in area.spaces:
            if space.type == 'VIEW_3D':
                sh = space.shading; ov = space.overlay
                sh.type = 'SOLID'
                sh.light = 'FLAT'
                sh.show_backface_culling = True
                sh.background_type = 'VIEWPORT'
                sh.background_color = (0.28, 0.28, 0.28)
                sh.color_type = 'SINGLE'
                sh.single_color = (0.92, 0.92, 0.92)
                sh.show_cavity = True
                sh.cavity_type = 'BOTH'
                sh.curvature_ridge_factor = 0.25
                sh.curvature_valley_factor = 2.0
                sh.show_xray = False
                ov.show_wireframes = True
                ov.wireframe_opacity = 1.0
                ov.show_floor = False
                ov.show_axis_x = ov.show_axis_y = ov.show_axis_z = False
                ov.show_overlays = True
        break
```

### `material` — цветовая идентификация деталей

**Когда:** проверка принадлежности детали, цветовая дифференциация после сборки, крупные планы стыков.

```python
import bpy
for area in bpy.context.screen.areas:
    if area.type == 'VIEW_3D':
        for space in area.spaces:
            if space.type == 'VIEW_3D':
                sh = space.shading; ov = space.overlay
                sh.type = 'MATERIAL'
                sh.use_scene_lights = False
                sh.use_scene_world = False
                sh.show_backface_culling = True
                sh.show_xray = False
                if hasattr(sh, 'studiolight_intensity'):
                    sh.studiolight_intensity = 1.15
                sh.show_cavity = True
                sh.cavity_type = 'BOTH'
                sh.curvature_ridge_factor = 0.35
                sh.curvature_valley_factor = 1.4
                ov.show_wireframes = True
                ov.wireframe_opacity = 0.42
                ov.show_floor = False
                ov.show_axis_x = ov.show_axis_y = ov.show_axis_z = False
                ov.show_overlays = True
        break
```

### `realism` — максимальный реализм (презентация)

**Когда:** итоговый скриншот для оценки внешнего вида, демонстрация пользователю. Перед применением — пересчитать нормали.

```python
import bpy, bmesh

# Пересчёт нормалей (если были инвертированы после boolean)
for obj in bpy.data.objects:
    if obj.type != 'MESH':
        continue
    bm = bmesh.new(); bm.from_mesh(obj.data)
    bmesh.ops.recalc_face_normals(bm, faces=bm.faces)
    bm.to_mesh(obj.data); bm.free(); obj.data.update()

for area in bpy.context.screen.areas:
    if area.type == 'VIEW_3D':
        for space in area.spaces:
            if space.type == 'VIEW_3D':
                sh = space.shading; ov = space.overlay
                sh.type = 'MATERIAL'
                sh.use_scene_lights = False
                sh.use_scene_world = False
                sh.show_backface_culling = True
                sh.show_xray = False
                if hasattr(sh, 'studiolight_intensity'):
                    sh.studiolight_intensity = 1.2
                ov.show_wireframes = False
                ov.show_overlays = False
        break
```

### `xray` — просмотр внутренней геометрии

**Когда:** отладка полостей, проверка заливки, просмотр скрытых деталей (труба внутри корпуса, гнёзда магнитов).

```python
import bpy
for area in bpy.context.screen.areas:
    if area.type == 'VIEW_3D':
        for space in area.spaces:
            if space.type == 'VIEW_3D':
                sh = space.shading; ov = space.overlay
                sh.type = 'SOLID'
                sh.show_xray = True
                sh.xray_alpha = 0.25          # прозрачность: 0.1 (почти невидимо) … 0.5 (полупрозрачно)
                sh.color_type = 'MATERIAL'
                sh.show_backface_culling = False
                sh.background_type = 'VIEWPORT'
                sh.background_color = (0.22, 0.22, 0.22)
                ov.show_wireframes = True
                ov.wireframe_opacity = 0.3
                ov.show_floor = False
                ov.show_axis_x = ov.show_axis_y = ov.show_axis_z = False
                ov.show_overlays = True
        break
```

---

| Режим | Команда пользователя | Лучший для |
|-------|---------------------|------------|
| `tech` | «включи tech» | геометрия, размеры, стыки, резьба |
| `material` | «включи material» | цвета деталей, крупные планы |
| `realism` | «включи realism» | презентация, итоговый вид |
| `xray` | «включи xray» | полости, заливка, скрытые детали |

**Автовыбор режима агентом:**
- Шаги 4–6 (построение, проверка геометрии) → `tech`
- Шаг 7 кадр A → `tech`, кадр B → `material`
- Итоговая демонстрация / презентация пользователю → `realism`
- Отладка полостей, нормалей, внутренних деталей → `xray`

## Параллельная работа Cursor и Claude Code

Оба агента выполняют полный флоу через MCP и могут работать одновременно. Чтобы не затирать правки друг друга:

**Перед Шагом 3 (правка кода):**
- Перечитать `parts/{name}.py` и `config.py` — другой агент мог изменить файл.
- Если файл изменился с момента последнего чтения — применять свои правки поверх актуальной версии, не поверх старой.

**Перед Шагом 4 (отправка в Blender):**
- Перечитать все файлы, которые будут импортированы в Blender-скрипте.
- `del sys.modules` обязателен — Blender кеширует модули между запусками скриптов.

**Правило одного объекта:**
- Два агента не должны одновременно изменять один и тот же `parts/{name}.py`.
- Если задачи на одну деталь — выполнять последовательно, не параллельно.

# DiaBox — обязательный workflow

## ОБЯЗАТЕЛЬНЫЙ ПОРЯДОК ДЕЙСТВИЙ ПРИ ЛЮБОЙ ЗАДАЧЕ

### Шаг 0. Определить активную модель, режим и прочитать правила модели

Прочитать `.env` → взять `CURRENT_MODEL` и `MODE` → работать только в `models/{CURRENT_MODEL}/`.

**Как получить значения `.env` в коде:**

```python
import sys, os
# Корень проекта берётся из PROJECT_ROOT в .env (без хардкода пути)
# В Blender-скриптах используй: _root = env.project_root
sys.path.insert(0, os.environ['PROJECT_ROOT'])

from config.env import load as load_env
env = load_env()
# env.project_root   — абсолютный путь к корню проекта
# env.current_model  — например "rocket"
# env.mode           — "default" или "cut"
# env.debug_labels   — True / False
```

`config/env.py` — единственный правильный способ читать `.env` в Python. Не использовать `os.environ`, `dotenv` или `read_env` напрямую.

**Обязательно прочитать** `models/{CURRENT_MODEL}/ai/rules.md` — там специфика конкретной модели: форма, стек деталей, особенности режима cut для этой модели. Общие правила workflow дополняются этим файлом.

**Обязательно прочитать** `models/{CURRENT_MODEL}/config.py` — перед любой задачей:

1. Выписать все `*Config`-классы и их поля, относящиеся к затронутым деталям.
2. Понять, какие параметры уже есть, какие вычисляются через `derive_all()`.
3. Если для задачи нужен параметр, которого ещё нет — **добавить его в нужный `*Config`** (не хардкодить в `parts/`). Сообщить пользователю о новом поле.
4. Если задача касается внешней / покупной детали (линза, стекло, плата, батарея) — убедиться, что у неё есть свой `*Config` с физическими размерами. Если нет — создать. Размеры внешних деталей не хардкодить в `parts/*.py`.

**Режим `MODE` — критически важен для понимания геометрии:**

| MODE | Что видно в Blender | Что реально в коде |
|------|--------------------|--------------------|
| `default` | Полная модель | Полная модель |
| `cut` | Половина модели (разрез X=0) | **Полная модель** — `bisect` применяется деструктивно только в сцене, после `main()` |

**Режим `cut` — только инструмент просмотра, не конечный вид детали.**

Пример: задача «залить кольцо» → написать код, который строит **полное кольцо** (360°). После `main()` режим cut разрежет его пополам автоматически — и в сцене будет видно полукольцо залитое. Именно так выглядит правильный результат в cut-режиме.

**ЗАПРЕЩЕНО при `MODE=cut`:**
- Считать половину модели / полукольцо / полудеталь конечной геометрией
- Добавлять «заглушки», «перегородки», «дно» для закрытия среза — это артефакт отображения, не конструктив
- Делать `get_object_info` и считать x_min/x_max отражающими реальный диаметр (bounding box будет вполовину)
- **Адаптировать геометрию кода под половину модели**: любые полукольца, полудуги, ограничения по углу (например, `start_angle=0, end_angle=180`), условия `if x >= 0` и прочие «половинчатые» конструкции в `parts/*.py` — строго запрещены. Код всегда строит полную (360°) деталь. Разрез X=0 — исключительно инструмент визуальной проверки результата, он применяется автоматически после `main()` и не должен влиять на логику построения геометрии.

**При `MODE=cut` все размеры проверять так:**
- Высота (`z_max - z_min`) — корректна
- Диаметр: `x_max * 2` (т.к. x_min ≈ 0, деталь обрезана по X=0)

### Шаг 1. Анализ задачи

Перед любым кодом — понять контекст:

- Какие детали затрагивает задача? (`parts/*.py`, `config.py`)
- Какова роль каждой затронутой детали в конструкции? (назначение, материал, посадки)
- Какая геометрия должна получиться? (форма, размеры, стыки с соседними деталями)
- Есть ли зависимости между деталями? (стыковочные радиусы, пазы, резьба)
- Какие **чужие конфиги** нужно читать в этой детали? (посадка → конфиг ответной детали, чаша → конфиг вставляемой детали)

**Проверка гибкости перед кодом (обязательно):**

Для каждого числа, которое собираешься написать в `parts/*.py`, задать вопрос: «Что произойдёт, если пользователь изменит смежный параметр конфига?»

- Если размер зависит от другой детали — **брать из её конфига**, не дублировать.
- Если это конструктивный отступ / толщина / зазор — **добавить поле в конфиг**, не хардкодить.
- Если это вычисляемая величина (например, z-позиция стыка) — **формула из конфигов**, не число.

Цель: изменение любого одного параметра в `config.py` не должно ломать геометрию остальных деталей и не требовать ручной правки в `parts/*.py`.

**Мысленная проверка сквозного пересчёта (обязательно перед кодом):**

Для каждого параметра, от которого зависит задача, прогнать цепочку:

> «Если это значение изменится → что сдвинется в соседних деталях → они корректно подстроятся?»

Примеры опасных мест:
- Высота базовой секции изменилась → верхняя деталь должна подняться (через `derive_all`, не через число)
- Толщина стенки изменилась → примыкающие детали расширились → пазы в ответных деталях приняли новые радиусы
- Размер вставляемого элемента изменился → чаша / паз под него пересчитался автоматически (через конфиг этого элемента)

Если цепочка рвётся — в коде есть хардкод; найти и заменить формулой до отправки в Blender. Полный справочник межконфигурных связей — `ai/rules/config-conventions.md`, раздел «Межконфигурные зависимости».

Справка по деталям: `models/{CURRENT_MODEL}/docs/tech.md`.

**Для составных деталей и новых профилей — перед кодом описать алгоритм словами:**

Для каждой подчасти (`_build_*`, `_attach_*`, `_cut_*`) записать:
1. **Форма**: что это геометрически (труба, диск, цилиндр, кольцо с пазом...)
2. **Профиль `pts`**: перечислить ключевые точки по порядку обхода и что каждая означает
3. **Стыки**: где и как эта подчасть примыкает к соседним (по какому радиусу, на какой высоте)
4. **Операция**: `revolve_profile` → самостоятельный объект, или UNION/DIFFERENCE к чему

Пример (соединительное кольцо):
```
Форма: сплошной диск (прямоугольное сечение)
Профиль: (ring_inner_r, z_bottom) → (ring_inner_r, z_top) → (profile_r, z_top) → (profile_r, z_bottom)
Стык: правый край = profile_r(spl, z_top) — вровень со стенкой тела, без зазора
Операция: revolve_profile → отдельный объект MESH_*_ConnRing
```

Это занимает 2–3 строки и предотвращает ошибки обхода и самопересечения до написания кода.

### Шаг 2. Прочитать нужные файлы

- `config.py` — всегда, если задача затрагивает параметры
- `parts/{file}.py` — если правится конкретная деталь
- Перечитать перед отправкой в Blender (мог измениться другой агент или пользователь)

### Шаг 3. Внести правку в код

- Числа — только в `config.py`, не хардкодить в `parts/`
- Новая деталь → отдельный файл в `parts/`, импорты относительные (`from .config import ...`)
- При изменении конструкции/параметров — обновить docstring и комментарии рядом с кодом

**Стандарт документирования файла детали (ОБЯЗАТЕЛЬНО):**

Каждый файл `parts/{name}.py` должен начинаться с модульного docstring по следующей схеме. Разделы заполняются по мере того, как информация становится известна — при создании или при правке. Если раздел неприменим, его можно опустить.

```python
"""
<Человекочитаемое название детали> (<имена Blender-объектов через запятую>).

Назначение:
  Что это физически, зачем существует, какую роль играет в сборке.

Конструкция (снизу вверх / изнутри наружу):
  1. <подчасть>: <форма, материал, геометрический принцип>
  2. <подчасть>: ...

Связи с другими деталями:
  - <деталь A>: <тип связи — стык, посадка, резьба, Г-паз, содержит внутри, крепится к...>
    Конфиг: <откуда берутся параметры — XxxConfig.поле>
  - <деталь B>: ...

Содержит / вмещает:
  - <внешний элемент>: <что это, как вставляется, конфиг — XxxConfig>

Технические особенности:
  - <особенность печати, материал, зазоры, порядок boolean-операций, что нельзя менять>

Режим cut:
  <что видно в cut-режиме, есть ли артефакты, на что обратить внимание>

Скриншот:
  <какие соседние объекты перекрывают эту деталь и их нужно скрывать для проверки>
  <рекомендуемый режим отображения: tech / material / xray>
"""
```

**Правила обновления:**
- При любом изменении детали — обновить затронутые разделы docstring.
- Если в процессе задачи выяснилась новая связь (деталь A влияет на деталь B) — добавить запись в «Связи» **обеих** деталей.
- Если внешний элемент не имеет своего `*Config` — создать его (см. Шаг 0, п. 4).

**Как использовать при анализе задачи (Шаг 1):**
- Прочитать docstring затронутых деталей → сразу понять связанные детали и конфиги.
- Раздел «Скриншот» → сразу знать, что скрывать и какой режим выбрать.

**Чеклист проверок перед отправкой в Blender (Шаг 4):**

```bash
python models/{CURRENT_MODEL}/test/check.py                     # всегда — кросс-деталевые + все parts/*.test.py
python models/{CURRENT_MODEL}/parts/{name}.test.py              # при работе над конкретной деталью
```

| Когда запускать | Что проверяет |
|-----------------|---------------|
| Перед каждым Шагом 4 | `test/check.py` — стыки, стеки Z, PCD, посадки + `test_config()` всех деталей |
| При работе над одной деталью | `parts/{name}.test.py` — локальные инварианты этой детали |
| После построения в Blender (Шаг 4А) | `test_mesh(obj)` — manifold, объём, высота, позиция |
| При любом изменении детали | Обновить `parts/{name}.test.py` если ожидаемые размеры изменились |

Полное описание структуры тест-файлов → `ai/rules/internals.md`, раздел «Тест-файлы деталей».

### Шаг 4. Отправить модель в Blender

**Выбрать режим сборки по масштабу задачи:**

#### Режим A: одна деталь (инкрементальный — предпочтительный)

Если задача затрагивает одну деталь — пересобирать только её, не всю модель.

```python
import sys, os

# Blender не читает .env из shell автоматически.
# Агент читает PROJECT_ROOT из .env ДО отправки кода в Blender
# и подставляет его значение в переменную _root напрямую:
_root = '<PROJECT_ROOT из .env>'   # агент подставляет значение перед отправкой
_model = '<CURRENT_MODEL из .env>' # агент подставляет значение перед отправкой
for _k in list(sys.modules):
    if _k in (_model, 'lib', 'config') or _k.startswith((_model + '.', 'lib.', 'config.')):
        del sys.modules[_k]
if _root not in sys.path: sys.path.insert(0, _root)
_models = os.path.join(_root, 'models')
if _models not in sys.path: sys.path.insert(0, _models)

import bpy
# Подставить нужные импорты из {CURRENT_MODEL}:
from {CURRENT_MODEL}.common import make_spline, setup_scene
from {CURRENT_MODEL}.profile import PART_BODY, world_profile_knots   # нужную часть
from {CURRENT_MODEL}.parts.body import build_body                     # нужную деталь

col = setup_scene()
spl = make_spline(world_profile_knots(PART_BODY))
body, ring = build_body(spl, col)

# Размеры сразу — до скриншота
def _bb(obj):
    xs=[v[0] for v in obj.bound_box]; zs=[v[2] for v in obj.bound_box]
    print(f"[BB] {obj.name}: h={( max(zs)-min(zs))*1000:.1f}mm  z=[{min(zs)*1000:.1f},{max(zs)*1000:.1f}]mm  x_max={max(xs)*1000:.1f}mm")
_bb(body); _bb(ring)
```

Преимущества: быстро, не трогает другие детали, print размеров в том же ответе.

**При ошибке в execute_blender_code — стоп.** Не переходить к Шагу 5. Прочитать traceback, найти строку ошибки, исправить, повторить Шаг 4.

**Диагностика по типу ошибки:**

| Ошибка | Причина | Действие |
|--------|---------|----------|
| `ImportError: No module named '...'` | Модули не сброшены или неверный путь | Проверить блок `del sys.modules`; проверить `sys.path` |
| `ModuleNotFoundError` после правки файла | Старый `.pyc` в кеше | Добавить `del sys.modules` для нужного пакета |
| `AttributeError: '...Config' has no attribute '...'` | Поле переименовано или удалено | Открыть `config.py`, найти актуальное имя |
| `ValueError` из `check.py` | Нарушено геометрическое ограничение | Читать сообщение `check()`, исправить формулу в конфиге |
| `ValueError` из `derive_all` / `_validate_local_profiles` | `z[0] != 0` или `z[-1] != height` в `profile_knots` | Привести последний узел профиля к `height` секции |
| Объект создан, но `vertices_count = 0` | Профиль `pts` вырожден (все r=0 или самопересечение) | Добавить `print(pts)` перед `revolve_profile`, проверить r |
| Тёмные пятна / вывернутые нормали | Неправильный обход `pts` | Перевернуть список: `pts[::-1]` и скорректировать стартовую точку |
| `RuntimeError: Calling operator ... context is incorrect` | `bpy.ops.*` без `temp_override` в Blender 5.0 | Обернуть в `with bpy.context.temp_override(...)` |
| Полупрозрачная перегородка на плоскости разреза | `pts[-1] == pts[0]` — дублирующее ребро | Убрать последнюю точку профиля |

#### Режим B: полная сборка

Когда задача затрагивает несколько деталей или нужна проверка стыков:

```python
import sys, os
_root = '<PROJECT_ROOT из .env>'   # агент подставляет значение перед отправкой
_model = '<CURRENT_MODEL из .env>' # агент подставляет значение перед отправкой
for _k in list(sys.modules):
    if _k in (_model, 'lib', 'config') or _k.startswith((_model + '.', 'lib.', 'config.')):
        del sys.modules[_k]
if _root not in sys.path: sys.path.insert(0, _root)
if os.path.join(_root, 'models') not in sys.path: sys.path.insert(0, os.path.join(_root, 'models'))

exec(open(f'{_root}/models/{_model}/{_model}.py').read())
main()
```

`_root` и `_model` — значения из `.env`, прочитанные агентом на Шаге 0 и подставленные в код перед отправкой.
**Причина:** Blender не наследует переменные окружения из shell, `os.environ` в Blender-скрипте пуст.

Использовать `execute_blender_code`. При ошибке — стоп, диагностировать по traceback, не переходить к Шагу 5.

### Шаг 5. Проверить геометрию через get_object_info (ОБЯЗАТЕЛЬНО)

**До скриншота** — вызвать `get_object_info` для каждой изменённой детали:

1. **Размеры**: `world_bounding_box` → высота и диаметр в мм. Формулы и паттерн `_print_bb` — `ai/rules/internals.md` (раздел «Bounding box из `get_object_info`»).
2. **Признаки брака**: если `vertices_count` равен 0 или `world_bounding_box` вырожден (все координаты одинаковые) — деталь не построилась, возврат к Шагу 3.
3. **Вычислить z-центр детали** для камеры: `z_mm = (z_min + z_max) / 2 * 1000`

### Шаг 6. Установить камеру на изменённую зону (ОБЯЗАТЕЛЬНО перед скриншотом)

**НИКОГДА не делать скриншот без предварительной установки камеры.**

Камеру направлять **именно на изменённую деталь/зону**, используя `z_mm` из Шага 5:

```python
from {CURRENT_MODEL}.common import set_viewport_camera

# z_mm — центр изменённой детали (из bounding box Шага 5)
# distance — подобрать по высоте зоны: высота_мм * 0.001 * 1.5, минимум 0.05
set_viewport_camera(z_mm=<z_centre>, distance=<distance>)
```

`{CURRENT_MODEL}` — значение из `.env`.

**Метки деталей (`DEBUG_LABELS=true`):** Empty-маркеры размещаются правее разреза (x > 0). Метки читаемы только при виде **спереди** — стандартный угол камеры `rotation_euler_deg=(75, 0, -90)`. При виде сзади метки окажутся за моделью.

Универсальная формула distance: `высота_зоны_мм × 0.001 × 1.5`, минимум 0.05.

Конкретные значения `z_mm` и `distance` для текущей модели — в `models/{CURRENT_MODEL}/ai/rules.md`.

Если задача затрагивает конкретную деталь — **не использовать вид всей модели**, только зону этой детали.

### Шаг 7. Сделать скриншот (выбрать оптимальный режим)

Выбрать **один режим** по задаче. Второй кадр — только если первый не даёт полной картины.

| Что проверяем | Режим |
|---------------|-------|
| Форма, размеры, стыки, резьба | `tech` |
| Принадлежность деталей, цвета | `material` |
| Итог для пользователя / презентация | `realism` |
| Полости, заливка, скрытые детали | `xray` |

Код пресетов → `ai/rules/mcp-blender-workflow.md`, раздел «Режимы отображения вьюпорта».

**Скрытие мешающих деталей:**

Если нужная зона перекрыта соседними объектами — скрыть их на время скриншота, затем восстановить:

```python
import bpy

# Скрыть мешающие объекты (имена подставить по текущей модели)
hidden = ["MESH_ModelName_Part", "MESH_ModelName_Part_SubPart"]
for name in hidden:
    if name in bpy.data.objects:
        bpy.data.objects[name].hide_viewport = True

# ... get_viewport_screenshot() ...

# Восстановить
for name in hidden:
    if name in bpy.data.objects:
        bpy.data.objects[name].hide_viewport = False
```

Когда скрывать — спросить пользователя или скрыть самостоятельно, если очевидно (например, верхняя деталь закрывает кольцо, корпус закрывает внутреннюю трубу). После скриншота **всегда восстанавливать** видимость.

Проверка: геометрия корректна, камера смотрит на нужную зону, режим позволяет однозначно оценить результат.

## Структура модели

```
models/{name}/
├── {name}.py              # точка входа, main()
├── config.py              # все числовые параметры (мм)
├── profile.py             # силуэт: стек Z, world_profile_knots, derive_all
├── common.py              # хелперы: set_viewport_camera, apply_material, булевы и др.
├── docs/tech.md           # описание деталей, назначение, сборка
├── ai/rules.md            # специфика модели: стек деталей, камера, резьба, cross-config
├── test/
│   ├── check.py           # ТОЧКА ВХОДА: кросс-деталевые + автодискавери parts/*.test.py
│   └── geom_test.py       # расширенные меш-утилиты (assert_wall_solid и др.)
└── parts/                 # детали: по одному файлу на деталь, экспорт build_*
    ├── {name}.py
    └── {name}.test.py     # тесты детали: test_config() + test_mesh(obj)
```

### `test/check.py` — единая точка запуска всех проверок

```bash
python models/{CURRENT_MODEL}/test/check.py
```

Содержит кросс-деталевые проверки (стыки, стеки высот, PCD соединений, совместимость посадок) и автоматически запускает `test_config()` из каждого `parts/*.test.py`. Новые детали подхватываются без правки `check.py`. Полное описание архитектуры тестирования → `ai/rules/internals.md`.
